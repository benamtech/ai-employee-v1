# Generative UI Frontier

Status: active frontier track  
Updated: 2026-07-19  
Purpose: define the highest-upside UI work that is source-wired but not live-proven

## Current Position

Generative UI is likely AMTECH's most powerful long-term interface idea: software surfaces appear from the work Avery is doing instead of forcing the owner through fixed SaaS screens. The code has the right seams, but the idea has not been live-tested against a funded provider-backed Hermes loop.

This frontier does not replace or block the baseline normal-employee launch gate. First prove the canonical public onboarding, runtime, provider reply, connected-tool, approval, receipt, recovery, and deployment path. Then run the first provider-backed generative-UI acceptance slice on that accepted stack.

The grounded disposition of the continuous-field, CSG, psychographic, WebGPU, agency-gradient, and chain-of-thought concepts is recorded in `08-speculative-ui-research-disposition.md`. That record is part of this frontier's scope boundary.

## What Exists

- Typed work descriptors can carry work views such as table, schedule, diff, and form.
- Manager compiles owner-safe views into MCP-UI `ui://` resources.
- The web owner surface renders those resources in a sandboxed iframe.
- Iframe intents return to the host and route through existing approval/respond handlers.
- Unit tests prove escaping, `ui://` resource creation, approval-id binding, and fallback behavior.
- The owner surface already consumes a versioned context manifest, operating state, and deterministic adaptive layout plan.
- A research-employee fixture now exercises question, source trail, competing claims, active monitoring, delegated source review, synthesis, contradiction, and evidence.

Primary source and local proof:

## What Is Missing

- A live LLM/Hermes run that creates a useful view from real business context.
- Provider-backed proof that the generated surface maps to the correct approval/action and resulting external receipt.
- Rich interaction beyond basic accept/reject/respond.
- Complete visual alignment of generated UI with the Avery-first system.
- A human-readable explanation of why a generated surface appeared and what Avery will do with it.
- Production-server browser evidence for the fixture matrix.
- Accessibility and keyboard evidence beyond source-level foundations.
- Persistent explicit owner preferences for density and explanation depth.

The current Manager template still uses an independent legacy dark-mode media query and blue primary action. That is a source-level visual mismatch with the canonical light-only, AMTECH-red design system, not a reason to weaken the typed-template boundary.

## Product Rule

Generative UI must never mean raw model HTML. AMTECH owns the template, action grammar, approval gates, sandbox, and proof. The LLM may propose structured work; Manager compiles it into a safe surface.

Adaptive presentation is also bounded. It may use exact assignment, explicit owner experience/density, typed risk and work state, owner-safe context, viewport, and accessibility preferences. It may not infer emotion, personality, fatigue, vulnerability, or persuasion susceptibility; expose private reasoning; or silently reshape authority.

## Tuesday Boundary

The following are release-relevant now:

1. deterministic focus and region ordering independent of input order;
2. explicit prominence for high-risk decisions, returned active saves, and material delegated failures;
3. production-build browser testing, target-size and reduced-motion checks, and immutable evidence;
4. visible rationale/provenance and stable registered components;
5. preservation of the normal-employee production path.

The following are not Tuesday work:

- WebGPU reaction-diffusion rendering;
- psychographic or sentiment adaptation;
- CSG/SDF as the information architecture;
- gradient-flow navigation or probabilistic attention control;
- chain-of-thought trajectory inspection;
- autonomous layout learning against opaque reward functions.

## Acceptance Gate

The frontier becomes accepted only after a provider-backed Hermes run:

1. receives real or production-like business context;
2. emits a typed work view through Manager;
3. renders in owner UI without fixture data;
4. routes an owner action through the same approval/proof path;
5. leaves auditable proof IDs that bind the view, approval, action, external result, and exact SHA.

## Exact Next Sequence

1. Preserve the current typed `WorkView` grammar, Manager compilation, sandbox, approval-id binding, escaping, and text-card fallback.
2. Close deterministic-layout, accessibility, and production-build browser evidence gaps without changing authority semantics.
3. Align the Manager-owned HTML template to the light Avery design system: AMTECH red primary action, no dark-mode branch, accessible focus/contrast, and stable mobile sizing.
4. Add owner-readable explanation metadata: why this surface appeared, which business facts Avery used, what approval is requested, and what action will follow.
5. After the normal-employee production gate passes, run one narrow provider-backed slice from real business context to typed view to owner action to external proof, retaining every proof ID.
6. Only after that acceptance slice, expand interaction, direct manipulation, or learned adaptation.
