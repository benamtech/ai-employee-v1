# Speculative UI Research Disposition

Status: active production decision record  
Date: 2026-07-19  
Branch: `employee-production-tuesday`  
Purpose: preserve the useful ideas in the speculative UI research without putting unproven mechanisms on the Tuesday launch path

## Decision rule

The Tuesday candidate has at most 20 remaining UI-development hours. A concept enters that path only when it:

1. closes a standing owner-facing defect or release-evidence gap;
2. composes with the existing typed operating surface, Manager authority, Hermes runtime, and durable proof path;
3. is deterministic, accessible, inspectable, bounded, and testable on the exact candidate SHA;
4. does not infer sensitive or unverifiable user traits;
5. can fail without hiding work, changing authority, or destabilizing the normal-employee deployment path.

The existing `OperatingContextManifest`, `OperatingSurfaceState`, `AdaptiveLayoutPlan`, registered work objects, signed Review, and MCP Apps boundary remain the production architecture. The research is evaluated against that architecture, not used as permission to replace it.

## Disposition matrix

| Concept | Decision | Production interpretation |
|---|---|---|
| Agency as a continuum | Adapt now | Represent agency explicitly through authored states: employee working, owner decision required, held for approval, delegated, blocked, repairable, and complete. Never infer a hidden scalar from behavior. |
| Psychographic Vector State | Reject for product use | Do not infer sentiment, attention, fatigue, expertise, or approval propensity from interaction telemetry. Use explicit owner-selected experience and density preferences plus bounded task/runtime facts. |
| Continuous Interface Field | Defer as research metaphor | The useful property is smooth prioritization among stable regions. Implement this with deterministic scores and registered components, not a continuously simulated field. |
| Reaction-diffusion / WebGPU field engine | Reject for Tuesday | It adds GPU/browser variability, energy and performance cost, inaccessible motion risk, and no release-critical capability. Static CSS and bounded transitions are sufficient. |
| CSG / signed-distance composition | Defer | The safe near-term substitute is typed region composition with explicit precedence. SDF rendering may be explored later for visualization, never for authority or information architecture. |
| Risk-shaped governance boundary | Adapt now | High-risk decisions receive stronger deterministic prominence and visible consequence. Approval semantics remain Manager-owned and unchanged. |
| Gradient-flow navigation | Reject as navigation model | Users need stable landmarks, visible controls, keyboard order, and a reliable return path. Adaptation may reorder bounded regions but cannot turn navigation into an inferred attention field. |
| Fokker-Planck attention model | Reject | Attention is not an authority signal and cannot be reliably inferred from screen behavior. Validate comprehension and task completion directly instead. |
| Hamilton-Jacobi-Bellman optimization | Defer to offline research | Runtime optimization against a synthetic cost function would be opaque and hard to validate. Use explicit pass/fail vectors and measured usability outcomes. |
| Runtime heartbeat as visual oscillator | Reject | Polling and transport cadence are infrastructure decisions, not perceptual animation primitives. UI updates follow meaningful durable state transitions. |
| Subagents as coupled oscillators | Reject as implementation model | Delegation remains bounded parent/child work with purpose, state, result, and proof. Coordination is driven by dependencies and durable commands, not phase synchronization. |
| Business brain as attractor network / dreaming | Defer | Memory consolidation and recommendation research must remain outside the Tuesday UI boundary and cannot silently alter owner facts or policy. |
| Chain-of-thought trajectory anomaly detection | Reject | Private reasoning is not a browser artifact, acceptance signal, or scheming-detection surface. Use observable actions, receipts, policy violations, and outcome evaluation. |
| Living, morphing website | Adapt narrowly | Regions may appear, collapse, or reorder from typed state. Stable labels, focus, reading order, control placement, and reduced-motion behavior must remain predictable. |

## Production adaptation contract

Adaptive UI may use only:

- exact assignment and current authority state;
- explicit `owner_experience` and `preferred_density` settings;
- typed work state, risk, return conditions, failures, receipts, and bounded event counts;
- owner-safe runtime, connector, and context signals already compiled by Manager;
- viewport and accessibility preferences supplied by the browser;
- stable context fingerprints and registered rationale codes.

Adaptive UI must not use:

- inferred emotion, personality, fatigue, vulnerability, or persuasion susceptibility;
- private memory, raw provider payloads, secrets, soul files, AGENTS/CODEGRAPH source, or chain-of-thought;
- cursor surveillance, tab-switch scoring, approval-rate profiling, or sentiment-derived visual pressure;
- arbitrary model-authored DOM, CSS, scripts, actions, prices, claims, or links;
- continuous animation as the only representation of state.

The same bounded input must produce the same layout plan. Input order must not create layout jitter. High-risk owner decisions, returned active saves, failed delegation, blocked work, ambiguous effects, and revocation must remain visible before high-volume low-risk activity.

## Tuesday implementation scope

### P0 — do not expand the UI until these release boundaries are honest

1. Keep the canonical normal-employee deployment and fixture-free production acceptance path ahead of generative-UI claims.
2. Do not present fixtures, development-server screenshots, source wiring, or old provider packets as production proof.
3. Preserve exact assignment, approval, C3, receipt, and revocation semantics for every UI action.
4. Keep the public estimator visibly non-canonical and outside release acceptance.

### P1 — highest-value UI work for the remaining budget

1. Run the fixture browser matrix against a production build, not only `next dev`.
2. Make adaptive focus selection deterministic across input ordering.
3. Promote high-risk decisions, returned active saves, and material delegated failures through explicit rationale codes.
4. Enforce 44px critical controls, visible unobscured focus, no mobile overflow, and reduced-motion-safe transitions.
5. Expand browser evidence to include the research fixture, degraded/blocked state, keyboard operation, target sizes, and production-server metadata.
6. Record immutable source, browser, accessibility, and layout reports against the exact SHA.

### P2 — release-adjacent, not Tuesday launch blockers

1. Provider-backed typed generative work object from real business context through approval and external receipt.
2. Proof search and job/customer/action filtering.
3. Persistent owner-selected density and explanation preferences.
4. Broader public, billing, portal, artifact, and admin visual alignment.
5. Direct manipulation, undo/revert, offline job-site capture, and richer MCP Apps.
6. Controlled usability experiments comparing authored and adaptive layouts with task success, error, recovery, comprehension, and trust measures.

## Acceptance boundary

This record does not promote any research concept to production acceptance. The Tuesday UI candidate remains accepted only through the canonical UI validation gates, a production build, browser/accessibility evidence, and fixture-free normal-employee proof on the exact deployed SHA.
