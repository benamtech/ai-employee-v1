# Live Employee: Hermes Runtime Contract + Event Bus (Phases 3 / 3A / 4-core)

## Context

The audit (memory: `mvp-build/memory/2026-07-03-1211-architecture-audit-three-critical-systems.md`) found the Manager's entire Hermes client (`apps/manager/src/lib/runtime.ts`, 47 lines) calls two invented endpoints (`/messages`, `/events/work`) that no Hermes build serves — so no message has ever round-tripped to a real employee, and the event bus, while having real signature verification and dedupe, has no importance routing, a dead adapter registry, an off-rail Twilio inbound, and a presence-blind delivery tail. This plan makes the employee live and event-driven in one build: rewrite the runtime client against the **real** Hermes API (researched live from NousResearch/hermes-agent v2026.7.1 source + docs), promote the generic ingress spine (adopting `wiki/MVP/build-plan-current/phases/phase-03-implementation-plan.md` v2 with corrections), build the minimal Channel/Session/Presence router (Phase 3A), and land the first live wake path (Phase 4 core: Gmail reply -> live Hermes turn -> employee-authored validated `WorkEventDescriptor`). Founder confirmed scope: include live wake AND minimal router.

## Ground truth: the real Hermes API (drives all design)

1. API server is a gateway platform inside the `hermes gateway` process; **env-config only** (`API_SERVER_ENABLED/KEY/PORT/HOST` in profile `.env`; no config.yaml support). Bearer = per-profile `API_SERVER_KEY`. Health `GET /health`; feature detection `GET /v1/capabilities`.
2. Canonical conversation = REST session: `POST /api/sessions` (choose id; 409 `session_exists` tolerable), `POST /api/sessions/{id}/chat` = one synchronous turn (`message`, optional `system_message`), returns text + usage. `/chat/stream` SSE and Runs API (`POST /v1/runs` + events SSE) exist — Phase 5 seam.
3. **No structured output support anywhere** (verified in source). Descriptors = prompt-for-JSON + parse + validate + retry.
4. **No per-session lock on REST /chat** — the Manager must serialize turns per employee.
5. Session keys are per-platform (`agent:main:{platform}:...`) — SMS gateway, API, cron, webhook sessions never share a conversation. Manager-owned unification into one canonical API session is the correct architecture (validates our Twilio-bypass pattern).
6. Jobs HTTP API (`/api/jobs`) exists — scheduler seam only. Hermes webhook adapter (port 8644) runs fresh untrusted sessions delivering to chat platforms — considered and rejected as the event ingress; Manager-owned spine stays.

### Stage C — First live wake path (last; integrates A+B+D)

- **WI-14 (L) `apps/manager/src/lib/wake.ts` + atomic claim.** `wakeEmployeeForDescriptor(db, employeeId, payload)` = one queue entry containing up to two `chatTurn`s into the canonical session with a `system_message`:
  - Prompt: "INTERNAL WORK EVENT — not a message from your owner. Do not converse." + EVENT_TYPE + FACTS (safe fact JSON) + demand EXACTLY ONE fenced ```json block with `{move, title (<=60), summary, deliverable? {type, title, refs, leaves_business, money, reversible, acceptance}, suggested_next_action}`; rule: email/money next-steps MUST include a deliverable with `approve`; never invent ids. Schema deliberately **omits** identity fields — Manager stamps `account_id`/`employee_id`/`source_event_id`/`proof` after parsing (identity never model-authored).
  - Extraction: last fenced ```json block -> parse; fallback balanced-brace scan. Then stamp identity + `validateWorkEventDescriptor`. Retry policy: exactly one retry in-session with the validation error; second failure -> `enqueueRepair("wake_descriptor_invalid:<reason>")`, owner sees nothing.
  - Restructure `deliverEmployeeEvent` wake branch: insert `inbound_events` row `status:"claimed"` via `insertDedup` **before** waking (0010 unique key = at-most-one wake under at-least-once webhooks) -> wake -> success: `bindApprovalIfNeeded`, merge descriptor, `status:"received"`, route via router; failure: `status:"repair"` + repair row. `deliver_only` keeps today's flow.
